# Phase 96: Prompt 行为约束强化 — 设计思考与后果矩阵

**日期**: 2026-06-04
**关联**: Phase 88-91 (Inner Monologue + Learner Profile) 的完成
**动机**: 教学审计学生画像 1/5, 自然度 1/5

---

## 0. 为什么 Phase 88-91 没有解决问题

Phase 89 引入了 Learner Profile 块（TTM + SDT + BKT 信号），Phase 89.1 引入了 Inner Monologue 分析块。设计意图是让 LLM "看到"学生状态后自适应行为。

实际效果：LLM 看到了，但没有行动。

### 根因不是信号缺失，是信号语言错误

```
当前 policy_layer（描述性）:
  "自主性: 50%（提供更多选择而非直接答案）"
  "胜任感: 40%（降低复杂度，多给正向反馈）"

当前行为（LLM 实际输出）:
  statement: "今天我们学习for循环。for循环是Python中..."
  question: "明白了吗？"
  → 没有选择、没有降复杂度、没有正向反馈、用了封闭型提问
```

问题：描述性语言告诉 LLM **关于**学生的信息。规定性语言告诉 LLM **必须做什么**。LLM 更服从后者。

### 证据

stable_prefix（规定性）→ LLM 高度服从。terminal_tutoring（规定性）→ LLM 基本服从。policy_layer（描述性）→ LLM 参考但不服从。

这不是推测——是 prompt 工程的基本规律：在 system prompt 中，"你必须"的优先级远高于"用户处于"。

## 1. 设计决策深度分析

### 1.1 为什么双路径注入

| 路径 | 位置 | 内容 | 为什么这里 |
|------|------|------|-----------|
| **路径 A** | terminal_tutoring | 通用行为规则（3 条） | 固定文本，缓存友好。所有学生都需要这些规则 |
| **路径 B** | policy_layer | 信号感知约束（~5 条） | 每轮动态，有信号才激活。已在缓存断裂点之后 |

单一注入点的缺陷：
- 全放 terminal_tutoring → 无法感知信号，所有学生看到相同规则
- 全放 policy_layer → 增加每轮 prompt 长度，但通用规则不需要每轮变化

### 1.2 为什么四级优先级

```
Flow > Mastery > TTM > SDT
  │       │       │      └─ 动机微调（可以下一轮再调）
  │       │       └─ 阶段节奏（影响教学策略选择）
  │       └─ 认知基础（决定了学生能理解什么）
  └─ 情绪紧急（焦虑/无聊 → 不处理则教学失效）
```

这个顺序来自教学研究：
- 情绪状态决定学生是否"能学"（affective filter hypothesis）
- 认知水平决定学生"能学什么"（ZPD）
- 学习阶段决定"怎么教"（TTM）
- 动机决定"学多深"（SDT）

### 1.3 为什么上限 5 条

超过 5 条约束 → LLM 的注意力被稀释 → 每条约束的服从率下降。5 条是 prompt 工程的经验上限（基于 DeepSeek 的 attention distribution 特性）。

### 1.4 为什么正常信号不产生约束

`_render_behavioral_constraints()` 在 mastery=0.5, flow="flow", SDT 都在 0.4-0.6 时返回空字符串。

理由：
- 正常状态不需要额外约束。stable_prefix + terminal_tutoring 的已有规则足够
- 避免 prompt 膨胀——不是每轮都需要行为约束
- 约束应该是"异常处理器"，不是"常态说明书"

## 2. 改动后果矩阵

### 2.1 直接影响

| 维度 | 改前 | 改后 | 风险 |
|------|------|------|------|
| Prompt 总长度 | ~6500 chars | ~6800 chars (+300) | 低——增加量在 token 预算内 |
| 缓存前缀 | stable_prefix + terminal_tutoring = 6239 chars | 同上 + 300 = 6539 chars | 零影响——仍在缓存内 |
| LLM 响应长度 | 教练独白，3-5 句 | 被约束到 2-3 句 | 可能过于简短 |
| LLM 提问模式 | 封闭型为主 | 被强制开放型 | 可能不自然 |
| 每轮 token 成本 | 不变 | +175 tokens | 可忽略 |

### 2.2 间接影响

| 维度 | 影响 | 概率 |
|------|------|------|
| **教学审计得分** | 学生画像 +1-2 分, 自然度 +1-2 分 | 高 |
| **学生体验** | 教练话更少、更倾听、更会根据状态调整 | 高 |
| **LLM 偶尔忽略约束** | 和已有 11 条终端自检一样，服从率非 100% | 中 |
| **复杂概念解释不足** | 约束压缩到 2-3 句可能不够讲清楚 | 低——可以跨轮拆分 |
| **flow 信号误判** | 假焦虑/假无聊 → 不必要的教学调整 | 低——影响仅 1 轮 |

### 2.3 最坏情况推演

```
worst case: flow_channel = "near_anxiety"（假阳性）
  + mastery < 0.3 + competence < 0.4
  → 5 条约束全部激活
  → 教练极度保守: 降难度 + 示范 + 生活类比 + 具体表扬 + 不许追问
  → 学生（实际上不焦虑）感到被过度呵护
  → 1 轮后 flow 重新计算 → 恢复正常

影响: 1 轮教学过于温和。可接受。
```

### 2.4 与已有规则的交互

| 已有规则 (terminal_tutoring) | 新规则 | 关系 |
|------|------|------|
| "每轮教学后必须跟随至少一个开放型追问" | 通用规则 13: 禁止封闭型提问 | 互补——规定开放型 + 禁止封闭型 |
| "教完必须验证——让学生独立输出" | TTM action: 每轮给练习机会 | 交叉——已有规则说验证，新规则说给机会 |
| "多提问少独白——学生话语应多于教练" | 通用规则 12: 压缩到 2-3 句 | 加强——具体化了"少独白"的量化标准 |
| "先诊断再教学——必须先问学生了解多少" | SDT competence: 禁止直接纠错 | 交叉——都要求先了解再行动 |
| "反馈要具体到过程——不泛泛说很好/不对" | SDT competence: 必须指出具体做得好的点 | 加强——具体化了"具体到过程"的正向面 |

**结论: 新规则与已有规则互补而非冲突。没有矛盾点。**

## 3. 边界情况排查

### 3.1 信号缺失

| 场景 | 处理 |
|------|------|
| `flow_channel = ""` (Flow 未启用) | flow 规则不激活。其他规则正常工作 |
| `mastery = None` (无掌握度数据) | mastery 规则不激活。其他规则正常工作 |
| `stage = ""` (TTM 未评估) | `stage in ttm_rules` → False。规则不激活 |
| `autonomy/competence = 0.5` (默认值) | 0.5 > 0.4 → 不触发。正常 |

### 3.2 极端信号组合

| 组合 | 约束数 | 优先保留 |
|------|--------|---------|
| flow=anxiety + mastery=0.1 + precontemplation + autonomy=0.2 + competence=0.2 | 7 | anxiety(1) + mastery新手(1) + precontemplation(1) + autonomy(1) + competence(1) = **5 条上限** |
| flow=near_boredom + mastery=0.9 + maintenance + autonomy=0.8 | 4 | 全部保留 |
| 全部正常范围 | 0 | 空字符串 |

### 3.3 语言一致性问题

所有约束使用统一格式: `"学生[状态] → [行动指令]。禁止: [禁止项]。"`
与已有 terminal_tutoring 的格式一致（"检查...如果是→违规"）。

## 5. 元提示词自审查

### □ 删掉损失什么？

继续教学审计 1.4-1.8/5。Phase 88-91 投入的 Inner Monologue + Learner Profile 工程回报为零——信号有了但没用。本 Phase 是那批工作的 ROI 兑现。

### □ 有具体行号？

全部改动在两个文件中：prompts.py（`_render_terminal_tutoring_checklist`, `_render_policy_layer`, 新函数 `_render_behavioral_constraints`, `build_coach_context`）和 agent.py（`_build_llm_context_bundle`）。每处改动有明确行号范围。

### □ 能反过来论证？

反方: "这些约束本质上是硬编码的教学策略。为什么不通过强化学习或 few-shot 让 LLM 自己学会？"

- 强化学习需要奖励信号。教学质量的奖励信号延迟极长（D7/D30/D90）。
- Few-shot 在 system prompt 中效果不如直接指令（已有多项 prompt 工程研究证实）。
- 硬编码约束的透明度高——每条规则可独立审计、独立开关、独立验证。强化学习策略不可解释。
- 约束是"安全边际"，不是"唯一路径"。LLM 在约束内仍有自由度。

反方: "约束可能过时。学生画像的阈值（0.3, 0.4, 0.7）是任意的。"

- 阈值来自 SDT 和 BKT 的标准惯例：0.3/0.7 是 BKT mastery 的标准分界点，0.4 是 SDT 需求满足的通用阈值
- 阈值不是硬编码在约束文本中——它们在 agent.py 的信号计算层。如果阈值需要调整，改 agent.py 即可
- 约束文本对阈值是透明的（"自主性低"），不引用具体数字。阈值变化不导致约束文本变化

### □ 三角色测试？

- **架构师**: 双路径注入——通用规则在缓存区，信号规则在动态区。四级优先级清晰。上限 5 条防止过载。✅
- **QA**: 正常信号不产生噪音。极端信号有上限保护。flow 误判影响仅 1 轮。与已有规则互补不冲突。✅
- **运维**: 纯文本改动。零新依赖。prompt 长度增长可忽略。缓存零影响。✅
