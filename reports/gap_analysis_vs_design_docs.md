## 三、偏离的根本原因分析

两份设计文档与当前代码库之间存在**系统性、根本性的定位偏差**：

| 维度 | 设计文档假设 | 当前代码库假设 | 冲突 |
|------|------------|--------------|------|
| 用户交互 | LLM 通过自然语言对话与用户互动 | 系统接收数值信号（engagement/stability/volatility → float [0,1]） | ❌ |
| 数据源 | 多源传感器（手机、手表、应用 API） | 无外部输入源 | ❌ |
| 优化目标 | 用户长期成长（学习迁移、创造性、独立思考） | 系统自身治理（是否允许升档/干预） | ❌ |
| AI 角色 | 教练/伴学（主动对话、引导、激励） | 门禁/审计（被动评估、阻断/放行） | ❌ |
| 记忆模型 | Letta 外部长期记忆（跨会话） | SQLite 事件哈希链（evidentiary） | ❌ |

## 四、代码存在的具体问题

### 4.1 中圈 L0/L1/L2 在教练场景中的适用性问题

- **L0Estimator** 接收 `engagement/stability/volatility` 三个信号——这些信号在两份设计文档中无处定义。文档对应的是 `dwell_time/behavioral_state/social_activity` 等数字表型指标。
- **L1Estimator** 的 `shock/memory/trend` 逻辑在纯规则法中过于简化，缺少文档要求的贝叶斯知识追踪（pyBKT）等概率模型支撑。
- **L2Estimator** 的 COM-B 映射（capability/opportunity/motivation）有理论参照，但缺少与 TTM 阶段的联动。

### 4.2 门禁引擎的隐性缺陷

- `_default_pass()` 方法在无输入时返回 pass（第 444 行），这在未连接真实数据源时**静默通过所有门禁**，可能掩盖配置错误。
- `_evaluate_7_framing_gate` 中 `passed = bool(audit_pass) is True`（第 391 行）——`bool(audit_pass) is True` 始终返回 False，此为**明显 bug**（应为 `bool(audit_pass) is True` 改成 `audit_pass is True` 或 `bool(audit_pass)`）。实际上由于未传递输入的 `_default_pass` 机制，此 bug 不影响当前行为，但一旦连接真实数据将失效。
- **确认**：`bool(x) is True` 在 Python 中确实始终为 `False`，因为 `bool(x)` 返回 `True` 或 `False`，而 `True is True` → OK，但 `True is True` 为 True，`bool(True) is True` 也是 True —— 等等让我重新检查。实际上 `bool(True) is True` => `True is True` => `True`。所以当 `audit_pass=True` 时，`bool(audit_pass) is True` => `True is True` => `True`，通过。而当 `audit_pass=False` 时，`bool(False) is True` => `False is True` => `False`，阻断。**这是对的**，但风格上有风险，应该用 `audit_pass is True` 或直接 `bool(audit_pass)`。

### 4.3 Overselling 风险

- `src/inner/gates/` 的 8 道门禁名称直接引用 V18.8 核心转向概念（Agency Gate, Learning Gate 等），**但没有任何数据源实际驱动这些门禁**。这造成"门禁骨架看似完整，实际功能为零"的工程债务。
- 同样的，`src/inner/no_assist/` 的评分逻辑虽然值得保留，但在当前管道中未与任何数据生产/消费路径连接。

### 4.4 缺失的关键基础设施

- 无 LLM API 调用、无 prompt 管理、无对话状态管理 → **系统无法与用户"对话"**
- 无任何传感器/设备 API 集成 → **无法获取文档要求的任何外部数据**
- 无向量数据库、无 embedding → **无法实现 Letta 风格的长期记忆**
- 无 SQLite 会话数据的 CRUD（仅有 append-only ledger）→ **无法存储用户画像**
- 无 cron/调度器 → **无法实现 JITAI 适时干预**

## 六、结论与路径建议

### 核心结论

1. **现有代码不是"不完整"，而是"做错了方向"**：它是一个治理/门禁系统，而非教练/伴学系统。698 个测试通过的代码在治理场景下是健康的，但对设计文档描述的教练场景贡献极小。

2. **V18.8 核心转向只在门禁骨架层面被简单反射，未落为运行时行为**：Agency Gate、Learning Gate 等的 engine 代码存在，但无任何数据源填充其输入参数，也无任何对话交互来驱动这些门禁的产生。

3. **最大缺口不是"增量开发"，而是"范式切换"**：要从当前的 "input: float signals → output: GO/WARN/FREEZE 治理" 切换到 "input: 用户对话+传感器数据 → output: 个性化教练策略" 是一个系统级重构，不能通过渐进修补完成。

### 建议的路线

| 优先级 | 行动 | 估算工作量 |
|--------|------|-----------|

| P1 | 若选择教练方向：搭建 LLM 对话层 + prompt 管理 + 基础向量记忆 | 3-5 天 |

| P2 | 构建 No-Assist 数据生产管道（实际用户作答→评估→落账本） | 1-2 天 |
| P2 | 实现双账本（Performance Ledger vs Learning Ledger 分离） | 2-3 天 |
| P3 | 实现主权脉冲 + 前提改写追踪 | 2-3 天 |
| P3 | 集成外部数据源（qs_ledger 风格的数据桥接） | 3-5 天 |

### 若维持当前治理门禁定位

则应找到与文档的新关系定位——将 8 道门禁作为教练系统的**质量保障子系统**，而非替代教练系统本身。在此情况下建议：
- 当前代码保持冻结作为治理基线
- 在两份设计文档指导下另起"教练对话层"项目
- 通过 No-Assist evaluator + 8 道门禁接口建立两系统之间的契约耦合
