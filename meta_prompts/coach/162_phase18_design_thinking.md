# Phase 18 Design Thinking — 研究管线：多实例并行 Agent 自优化调研系统

## 1. 问题定义

教学水平从 Level 3 提升到 Level 5 需要的不是零散的功能补丁，而是一次系统性的调研：
- 当前系统离家教级差什么
- 市场上有什么可借鉴的
- 学术界有什么方法论

但手工执行 8 个角色的调研（代码审计、数据审计、学术搜索、综述、可行性、路线图、评审、路由）需要 15-30 分钟一个人的工作，而且不可重复——每次评估都需要重新做。

需要一个可一键执行、自动收敛、自我改进的调研系统。

## 2. 架构决策

### 为什么用 Python 调度器 + LLM API 调用，而不是一个 LLM 角色扮演？

| 维度 | 角色扮演（一个 LLM） | 多实例并行（Python 调度器） |
|------|---------------------|--------------------------|
| 并行度 | 串行切换角色 | 可真正并行 |
| 状态保持 | 依赖 LLM 上下文窗口 | JSON 文件持久化 |
| 可重复性 | 手动操作 | `python run_*.py` 一键 |
| 错误恢复 | 断线重来 | 检查点恢复 |
| 自优化 | 无 | GEPA/SCOPE/MASPO |

### 为什么每个 Agent = 单次 LLM 调用？

自省迭代内化到一次生成中完成，而不是多次调用。LLM 可以在单次响应内做到"读代码 → 找发现 → 自省 → 标注关联 → 收敛"的完整循环。

## 3. 状态协议

shared_state.json 的结构设计原则：
- JSON 格式（Python 和 LLM 都可读写）
- findings 必须带 source_agent 标签
- findings 之间通过 related 字段建立关联关系
- 支持检查点恢复

## 4. 自优化设计

GEPA + SCOPE + MASPO 的融合方式：
- GEPA：用 LLM 分析执行记录中的弱点 → 生成改进指令
- SCOPE：跨运行记忆文件累积历史经验
- MASPO：优化单个 Agent 时考虑其与上下游的交互关系
- 经验注入：前次运行的经验自动追加到 prompt 末尾

## 5. 验收标准

- `python run_research_pipeline.py` => 0 错误退出
- 第二次运行自动使用第一次优化的 prompt
- scope_memory.json 累积了 ≥2 次运行的跨运行经验
