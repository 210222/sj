# Phase 16 Design Thinking — 能力唤醒

## 1. 问题定义

当前 Coherence 的教学系统有一个根本性的不对称：

**开发侧**：9 个高级模块全部实现并通过测试，配置在 `coach_defaults.yaml` 中 `enabled: false`。
**用户侧**：不知道这些模块存在，不知道打开有什么影响，甚至不知道 SettingsPanel 里有 toggle。

这不是代码缺失，这是"知情权 + 决策权"缺失。系统替用户做了"默认关"的决定，却没有把选择权交还给用户。

## 2. 目标体验

Phase 16 的目标不是加新的教学功能，而是加"知情的入口"：

- **首次对话时**：系统主动告知"我还有一些高级能力可以打开，你想看看吗？"
- **用户询问时**：解释每个模块的作用、影响、推荐场景
- **用户决定时**：说"打开 X"就自动修改配置，并告知影响
- **设置面板**：完整展示所有模块，不再是部分隐藏

## 3. 核心字段

### Capability 标准数据模型
```
capabilities:
  [module_key]:
    name: 中文显示名
    purpose: 作用说明（一句话）
    impact: 启用/禁用的影响（含正反两面）
    recommended_for: 推荐启用场景
    risk: 风险等级和说明
    auto_affects: 自动影响的关联模块
```

### Awakening 响应字段
- `_awakening`: { `triggered`: true, `capabilities`: [...] }
- 首轮用户返回，非首轮不返回
- 前端据此决定是否展示 AwakeningPanel

## 4. 依赖顺序

1. 能力目录先行（没有数据就无法展示和操作）
2. 后端检测（知道何时触发）
3. 前端 UI（知道怎么展示）
4. 对话启用（用户可通过聊天操作）
5. 设置面板补全（双通道）
6. 验证收口

## 5. 验收标准

- 新用户首轮对话中能看到能力列表
- 回访用户不重复看到
- 用户对话启用后模块真正工作
- SettingsPanel 9 个 toggle 全在
- 全量回归无退化

## 6. 风险

- 首轮检测不能破坏已有新用户的正常对话
- YAML 写入需原子性，写入失败时不能损坏配置
- "打开所有" 可能导致不兼容的模块同时启用
